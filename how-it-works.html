How It Works

System overview, reference guide, and troubleshooting
Dashboard How It Works
Data Flow
1. Bill arrives in Google Drive
A utility bill PDF is dropped into the designated Drive folder. All bill types (electricity, natural gas, water) go into the same folder. The system identifies the type automatically.
2. Apps Script detects the new file
A time-based trigger runs every 15 minutes. It compares files in the Drive folder against a log of already-processed file IDs. New files are queued for processing.
3. Claude API extracts all data fields
The full PDF is sent to the Claude API (primary extractor). Claude reads all pages and returns a structured JSON with: bill type, property ID, dates, usage, cost, estimation flag, demand, and demand cost.
4. Grok API independently validates
The same PDF is sent to the Grok API with identical instructions but without Claude's output. Grok reaches its own conclusions. This second read catches misreads before they reach the sheet.
5. Outputs are compared and validated
The system compares both outputs field by field. Numeric values must agree within 2% (to handle rounding). Dates must be exact. Sanity checks run on all required fields regardless.
6a. Agreement → written to Sheet
If both APIs agree and all sanity checks pass, the data is written to the correct tab (Electricity, Natural Gas, or Water) with a sequential row number. The Audit Log is updated.
6b. Disagreement → Review Queue + email alert
If the APIs disagree on any field, or sanity checks fail, the bill goes to the Review Queue tab. An email alert fires immediately with the row number, filename, which fields disagreed, and both extracted values.
7. Dashboard updated
After each run, the system writes an updated bills.json file to this GitHub repository. This dashboard reads that file and refreshes automatically every 60 seconds.

When a Bill Is Flagged
You will receive an email alert
Subject line: [Bill Parser] Review Required - Row #X - Electricity
The email includes the row number, filename, which fields disagreed, and both extracted values side by side.
  1. Open the email. Note the row number and which fields are in conflict.
  2. Open the original bill using the Drive link in the email.
  3. Look up the correct value directly from the bill for the flagged field(s).
  4. Open the Google Sheet → Review Queue tab. Find the row by its number.
  5. Manually enter the correct values into the appropriate utility tab (Electricity, Natural Gas, or Water) using the next available row number.
  6. Update the Review Queue row status from PENDING to RESOLVED.
Important: the system never writes wrong data
A bill sitting in Review Queue is always better than a wrong number written to the sheet. The queue is the intended safety net, not a failure state.
Security

All sensitive credentials are stored in Google Apps Script Script Properties — Google's built-in encrypted key-value store. No keys are written in the code or committed to this GitHub repository.

Keys stored in Script Properties
  • CLAUDE_API_KEYAnthropic API key for Claude extraction
  • GROK_API_KEYxAI API key for Grok validation
  • GITHUB_TOKENFine-grained PAT, repo scope only (contents: write)
  • GITHUB_REPOe.g. your-username/bill-parsing-system
  • SHEET_IDGoogle Sheet ID from the URL
  • DRIVE_FOLDER_IDDrive folder ID where bills are dropped
  • ALERT_EMAILEmail address for Review Queue alerts

To set or update a key: open the Apps Script editor → Project Settings → Script Properties. Keys can be rotated at any time without changing any code.

The GitHub token should be a fine-grained personal access token scoped to only this repository with only contents: write permission. It cannot access any other repos or account settings.

The GitHub repository should remain private. The dashboard (GitHub Pages) can be accessed via the Pages URL regardless of repo visibility if GitHub Pages is enabled on a public repo — keep the repo private and the dashboard will only be visible to those with the URL.

Rotating API Keys
  1. Generate the new key from the provider's dashboard (Anthropic Console, xAI Console, or GitHub Settings).
  2. Open the Apps Script editor for the Bill Parser project.
  3. Go to Project Settings → Script Properties.
  4. Find the relevant key, click Edit, paste the new value, save.
  5. Test by running diagnosConfig() from the editor to confirm the key is recognized.
  6. Revoke the old key from the provider's dashboard.
Adding a New Bill Type

When a new utility provider is added to the portfolio:

  1. Run a sample bill through the system. Check the Review Queue to see what fields Claude and Grok extracted.
  2. If extraction is wrong, open Claude.gs and Grok.gs. Add a provider-specific note in the extraction prompt describing the layout variation for the new provider.
  3. If a new column is needed, add it to the relevant tab header in Data.gs under HEADERS, and add it to the corresponding row array in writeToSheet().
  4. Commit the updated code to GitHub and copy the updated Code.gs into the Apps Script editor. No redeployment needed for time-based triggers.
  5. Test with one bill of the new type to confirm it processes correctly.
Glossary
kWh (kilowatt-hour)
Unit of electricity consumption. The Usage column for electricity bills is measured in kWh.
m³ (cubic metres)
Unit of volume. Usage for natural gas and water bills is measured in m³.
Demand (kW)
The peak power draw during a billing period, measured in kilowatts. Relevant for electricity bills only. Larger properties are billed separately for their peak demand in addition to total consumption.
Demand Cost
The charge associated with peak demand. Calculated as: Distribution + Transmission Network Charges + Transmission Connection Charges - Transformer Allowance (if applicable).
Ontario Electricity Rebate
A provincial credit applied to electricity bills. It appears as a "CR" (credit) and is subtracted when calculating final cost.
Estimation (Yes/No)
Whether the meter reading for this billing period was estimated by the utility (Yes) or based on an actual meter read (No). Estimated bills are flagged so they can be noted as provisional data.
Review Queue
The sheet tab where bills are sent when the two AI systems disagree or sanity checks fail. A bill in Review Queue has not been written to any utility data tab and requires manual entry.
Audit Log
A complete record of every bill processed by the system, including outcome (PROCESSED or REVIEW_QUEUE), timestamp, and whether Claude and Grok agreed.
Script Properties
Google Apps Script's built-in encrypted key-value store. Used to hold all API keys and configuration values. Accessed via Extensions → Apps Script → Project Settings.
bills.json
A data file in this GitHub repository written by the Apps Script after each processing run. This dashboard reads it to display current status. It contains summary counts, recent bill log, and Review Queue items.

Contact your system administrator. For urgent processing failures, check the Apps Script execution log: open the Apps Script editor → Executions (left sidebar) to see the last run output and any error messages.
